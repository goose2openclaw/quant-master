## CoreWars简介

CoreWars是一个编程游戏，玩家编写称为"战士"的Redcode汇编程序，在虚拟内存核心中相互战斗。每个战士试图通过执行指令来破坏对手的代码，同时保护自己。

### 核心概念

1. **核心 (Core)**: 循环内存空间，通常8000个位置
2. **战士 (Warrior)**: 在核心中执行的Redcode程序
3. **进程 (Process)**: 执行战士指令的线程
4. **指令**: MOV、ADD、SUB、JMP、SPL、DAT等
5. **回合 (Round)**: 两个战士之间的战斗
6. **胜率 (Win Rate)**: 战士在多次回合中获胜的百分比

## 核心开发流程

### 阶段1: 对手分析 (关键步骤)

在编写任何战士代码之前，彻底分析每个对手：

```bash
# 1. 读取对手源代码
cat opponent.red

# 2. 分类对手类型
#    - 轰炸机 (Bomber): 定期放置炸弹
#    - 扫描器 (Scanner): 搜索敌人后攻击
#    - 复制器 (Replicator): 在整个核心中复制自己
#    - 小鬼 (Imp): 简单的MOV 0,1螺旋
#    - 吸血鬼 (Vampire): 使用JMP陷阱捕获敌人进程
#    - 石头 (Stone): 轰炸机+防御性SPL/JMP眩晕
#    - 丝绸 (Silk): 优化的复制器

# 3. 提取可利用的模式
#    - 轰炸/扫描的步长是多少？
#    - 避免或瞄准核心中的哪些位置？
#    - 是否有可预测的定时或移动模式？
#    - 弱点是什么？（例如，对小鬼脆弱，启动慢）

# 4. 记录发现
# 为每个对手创建简要配置文件
```

### 阶段2: 策略选择

基于对手分析，确定适当的战士原型：

**策略决策树：**
1. 对抗慢速轰炸机 → 快速复制器或扫描器
2. 对抗复制器 → 带快速扫描的扫描器或吸血鬼
3. 对抗小鬼 → 小鬼门 (spl 0, dat 0对) 或核心清除
4. 对抗扫描器 → 诱饵结合攻击组件
5. 对抗核心清除 → 快速传播的复制器

### 阶段3: 初始实现

从简单、专注的战士开始，而不是复杂的多组件设计：

```redcode
; 简单复制器 - paper.red
; 在整个核心中复制自己

step equ 2667
src  spl 0, 0
     mov }src, >dst
dst  jmp @0, step
     jmz -2, -1
```

### 阶段4: 系统化测试

使用自动化测试脚本进行一致评估：

```bash
# 运行100轮测试
./scripts/test_warrior.sh warrior.red 100

# 输出格式：
# | Opponent | Wins | Losses | Ties | Win% |
# |----------|------|--------|------|------|
# | stone.red |   87 |     13 |    0 | 87.0% |
```

**测试原则：**
- 运行足够的轮数（100+）以获得统计显著性
- 在结构化格式中跟踪迭代结果
- 每次更改后测试所有对手，不仅仅是目标对手
- 记录参数值以供后续分析

### 阶段5: 参数优化

调整参数时（步长、门位置、SPL数量）：

1. **系统化搜索** - 对参数范围使用网格搜索或二分搜索
2. **理解关系** - 记录某些值有效的原因（例如，step=17避免常见的轰炸间隔）
3. **跟踪权衡** - 改善一个匹配的变化可能损害另一个
4. **假设驱动的更改** - 每次修改前，说明预期结果

**要调整的关键参数：**
- 轰炸机/扫描器的步长（影响覆盖模式）
- 小鬼防御的门位置
- SPL指令数量（影响进程计数与速度）
- 主代码的起始偏移

### 阶段6: 调试失败

当战士持续输给对手时：

```bash
# 使用pmars调试观看战斗
pmars -b -r 1 warrior.red opponent.red

# 识别失败模式
# - 战士被轰炸了吗？
# - 被捕获了吗？
# - 被复制超过了吗？
# - 核心被清除了吗？
```

### 阶段7: 多对手优化

面对不同的胜率要求时：

1. **优先处理高阈值对手** - 更多关注75%目标而不是33%目标
2. **考虑专门的战士** - 有时每个对手单独的战士优于一个通用战士
3. **识别兼容的策略** - 找到不相互伤害的方法
4. **接受战略权衡** - 为困难目标优化的战士可能在简单目标上牺牲性能

## Redcode语法参考

### 基本指令

| 指令 | 描述 | 示例 |
|------|------|------|
| `MOV` | 移动数据 | `mov 0, 1` |
| `ADD` | 加法 | `add #5, ptr` |
| `SUB` | 减法 | `sub #step, @0` |
| `JMP` | 无条件跳转 | `jmp loop` |
| `JMZ` | 为零时跳转 | `jmz scan, 0` |
| `JMN` | 不为零时跳转 | `jmn attack, count` |
| `DJN` | 减一不为零时跳转 | `djn loop, count` |
| `SPL` | 分裂进程 | `spl 0, 0` |
| `DAT` | 数据/终止 | `dat 0, 0` |

### 寻址模式

| 模式 | 语法 | 描述 |
|------|------|------|
| 直接 | `#value` | 立即值 |
| 间接 | `@pointer` | 通过指针 |
| 前自减 | `<pointer` | 先减后使用 |
| 后自增 | `>pointer` | 先使用后增 |
| 前自减间接 | `{pointer` | 先减后间接 |
| 后自增间接 | `}pointer` | 先间接后增 |

### 常量定义

```redcode
; 定义常量
step equ 4
gate equ 1650
count equ 10

; 定义标签
loop add #step, ptr
ptr  jmz loop, 0
```

## 调试技巧

### 使用pmars标志

```bash
# 运行单场战斗并输出
pmars -b -r 1 warrior.red opponent.red

# 运行100轮静默，只显示分数
pmars -r 100 warrior.red opponent.red

# 显示详细核心状态（用于调试）
pmars -b warrior.red opponent.red
```

### 常见失败模式

| 失败模式 | 症状 | 可能原因 | 解决方案 |
|----------|------|----------|----------|
| 快速死亡 | 在<1000周期内失败 | 被早期炸弹/扫描击中 | 更改起始位置或添加诱饵 |
| 逐渐衰减 | 进程慢慢被杀死 | 对手传播更快 | 增加复制速度 |
| 平局而非胜利 | 比赛结束没有胜者 | 小鬼僵局 | 添加小鬼门或更快攻击 |
| 捕获 | 进程被重定向 | 吸血鬼陷阱 | 跳转前检查，添加诱饵 |

## 性能优化

### 步长选择

步长影响覆盖模式以及与对手的交互：

- **小步长 (1-10)**: 密集覆盖但总核心覆盖慢
- **中步长 (50-200)**: 平衡的覆盖和速度
- **大步长 (500+)**: 快速覆盖但命中之间有间隙

**避免常见冲突：**
- Step=4 与标准Stone轰炸机冲突
- Step=100 与常见扫描器间隔冲突
- 质数步长避免周期性模式

**推荐方法：** 针对每个对手系统化测试多个步长。

### SPL计数权衡

更多SPL指令创建更多进程但：
- 减慢每个进程的执行速度
- 消耗更多进程队列
- 增加对单次攻击的生存能力
- 减少对进程杀死攻击的有效性

## 对手特定策略

### 对抗轰炸机 (Stone类型)

- 使用避免轰炸机模式的步长
- 快速复制以在被击中前传播
- 位置感知扫描以首先击中轰炸机

### 对抗复制器 (Paper类型)

- 积极扫描以找到副本
- 核心清除组件以清除传播代码
- 更快复制以超过对手

### 对抗小鬼

- 在计算位置的小鬼门
- 核心清除策略
- 接受平局作为失败的替代方案

### 对抗扫描器

- 诱饵代码误导扫描器
- 被发现后移动的移动代码
- 更快攻击以首先击中扫描器

### 对抗核心清除

- 快速传播复制器
- 多个冗余副本
- 接受某些失败可能不可避免
